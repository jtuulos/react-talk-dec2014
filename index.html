Flux

facebook.github.io/flux

Application architecture for building user interfaces

More of a pattern than a formal framework

  • Why should I try/use React?
  • Is this just another Backbone/Angular/Ember?
  • To answer that we need to go back
  • let's say websites are buildings
  • The Web Developer owns the land and the house
  • The DOM is the house staff, needs precise instructions and serves the guests
  • The User is the guest and expects good service
  • DOM is slow but reliable, between developer and user
  • the middle ages
  • stone castles, near impossible to rebuild (client-side)
  • the browser just lets people in and makes the place look nice – no interaction
  • developer can’t make allowances for user
  • specific buildings (addresses) for specific needs (map of town)
  • owner wants to have a big feast: can’t add more seats, can’t have a bigger kitchen
  • industrial age
  • long period with various technologies prospering
  • small victorian houses
    • improvement over time: plumbing, lighting, clocks, mechanical appliances
    • varied styles, built fast and crammed together
  • very DIY-driven for the owner (developer)
    • custom-built for the guest but with a lot of effort
    • still the same house, just different setup and furnishings
    • inefficient rebuilds and repaints
    • people still live in these old houses
    • jQuery is like a $10k gift certificate to IKEA, you can do a lot of stuff but everything comes from one place
  • wiring everything together is really hard to manage
  • backbone.tableSetter.js (connected to the table and chairs)
  • remember, jQuery is like IKEA?
  • jquery.FANTASTISK.js (some nice napkins)
  • jquery.JÄTTELIK.js (does a lot of things around the house)
  • antiqueOven.js (deprecated, incompatible with jQuery, kinda works)
  • Changing anything causes a messy cascade of other changes
  • you can have a plan (MVC framework) but you still need to tweak the DOM manually
  • modern times (and the future?)
  • The Dispatcher is like the architect, core part of React
  • what if everything could be re-built to accommodate every situation (state)?
  • major realization: data is fast, DOM is slow so have a separation between them
  • pretty much opposite to 2-way binding in Angular/Ember
  • user tells owner what they want to happen (actions)
  • owner forwards request to the architect who independently comes up with the best way to make changes
  • architect tells construction crew (DOM) what updates to make
  • the owner doesn’t have to worry about structural engineering anymore
  • the owner can focus on what the user needs
  • having a party?
    • 10 people? one dinner table. 1000 people? conference center
    • table and room magically expands when new guests are added
    • no people? the decorations won’t even be ordered
  • declarative UI
    • back to the old way of re-building everything
    • server-side rendering

Flux: One-Way Data Flow

It feels revolutionary.

No jQuery, no selectors

Virtual DOM >> DOM optimization

  • revolution implies making something important irrelevant
  • Facebook completely skipped the DOM while everyone else was creating complicated frameworks to manage it
  • no jQuery, no DOM-related libraries, almost no selectors
  • without jQuery you tend to use purpose-built micro-libraries
  • Controversial ideas like no direct DOM access and JSX
  • Facebook and Instagram (>1B MAU, non-trivial UI, developed in multiple teams)

also: GitHub Atom, Firefox, AirBnB, Reddit and more.

  • Great for data visualizations
  • Data-driven UIs? Not just React, D3.js also
  • Large scale apps spanning multiple teams (modularity, components)
  • Might be too much overhead for a quick single-page app
